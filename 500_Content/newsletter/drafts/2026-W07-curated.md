# 當 AI 開始寫出比你更像你的文字

> 2026-02-09 ~ 2026-02-15 — Dex 的週報

---

## 一、用 LLM 萃取你自己的「風格指紋」

這週我做了一件有點 meta 的事：把自己過去 78 篇 Threads 貼文丟給 LLM，讓它從 7 個維度分析我的寫作風格，產出一份 Style DNA。

效果超出預期。

它抓到的特徵比我自己能說出來的更精準——句式節奏、開場偏好、收尾習慣，全都有。接著我把這份 Style DNA 餵回去當 system prompt，用來生成新的草稿。

這形成了一個很有趣的迴圈：**用 AI 分析你 → 萃取風格 → 用風格生成新內容 → 你再微調**。

如果你有經營個人品牌，與其從零告訴 AI「我的風格是什麼」，不如直接丟 50 篇以上的歷史內容讓它自己歸納。你會發現 AI 比你更能說清楚「你是誰」。

---

## 二、Skill 不是工具，是知識管理的改革

這週看了一場 AgentSkill 的講座，有一句話直接改變了我對 AI agent 的理解：

**「多數人把 Skill 當 function call，但真正的價值在於將隱性知識結構化為可觸發的協議。」**

什麼意思？你平常怎麼寫 commit message、怎麼做 code review、怎麼規劃功能——這些藏在你腦子裡的「做事方法」，其實可以寫成 Skill，讓 AI 按照你的標準執行。

講座提到三種設計光譜：功能導向、個別導向、原則導向。大多數人停在第一層（「幫我格式化 JSON」），但真正強的是第三層——你定義原則，AI 自己判斷怎麼套用。

這跟寫 CLAUDE.md 或 system prompt 的邏輯完全一致：**你不是在教 AI 做一件事，你是在把自己的決策框架外部化。**

## 三、OAuth 是獨立開發者的隱形稅

為了串接 Threads API，我花了 20 分鐘跟 Meta 的 OAuth 搏鬥。

localhost redirect URI？封殺。HTTP？不行。HTTPS？也不行。Postman callback？還是不行。

最後解法是什麼？直接在 Meta Dashboard 用「用戶權杖產生器」手動產 token。一分鐘搞定。

這不是技術問題，是**大廠 OAuth 流程的設計假設你有公開 domain**。如果你是獨立開發者、在本地跑 side project，你根本不在他們的設計對象裡。

教訓很簡單：**動手之前先讀平台限制文件**。我如果先花 3 分鐘查文件，就不會浪費 3 輪無效嘗試。這個原則適用於所有第三方 API 整合。

## 四、AI 工具的隱形開銷：Token 管理

這週我把 CLAUDE.md 從 192 行壓縮到 70 行，省下 63% 的 token。

聽起來是小事，但算一下：這個檔案每次對話都會被載入。一天 10 個 session，一週 50 個，一個月 200 個。每次省下的 token 累積起來，就是真金白銀（或者至少是 rate limit 的緩衝空間）。

用 AI 工具的人很少想這件事——**你的 system prompt、你的 context window、你的指令檔，全都有隱性成本**。就像軟體工程裡的記憶體管理，你不處理它，它就慢慢吃掉你的效能。

這週我就因為多 session 切換 + 高 token 消耗，被 rate limit 反覆中斷，整個下午的內容管線卡住。

優化建議：定期 audit 你的 AI 指令檔，把能抽成 template 的抽出去，把過時的規則清掉。

## 結語

這週的四件事看似不相關——風格萃取、Skill 設計、OAuth 踩坑、Token 管理——但底層邏輯是同一個：

**把隱性的東西顯性化。**

你的寫作風格是隱性的，萃取成 Style DNA 它就變成可複用的資產。你的工作方法是隱性的，寫成 Skill 它就變成可觸發的協議。平台限制是隱性的，先讀文件它就變成可避開的地雷。Token 消耗是隱性的，量化它你才能管理它。

AI 時代最值錢的能力，可能不是寫 prompt，而是**把你腦子裡那些「理所當然」的東西結構化輸出**。

因為只有顯性化的知識，才能被 AI 放大。
