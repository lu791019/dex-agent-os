# 「超集模式」的工作流設計

## 核心論點
設計新流程時，不要取代舊流程，而是讓新的成為超集——舊流程自動變成子集，使用者永遠可以用自己習慣的粒度操作，這才是真正可持續演化的系統。

## 關鍵素材
1. **軟體版本相容性類比**：API v2 不砍 v1 端點，而是讓 v1 行為成為 v2 的子集——這就是 backward compatibility 的核心精神，同樣適用於個人工作流設計
2. **實際案例：daily-all vs daily-content**：`/daily-all` 是超集（包含同步、互動、內容產出），`/daily-content` 是子集（只跑內容），兩者共存、使用者按需選擇
3. **反模式：「大重構陷阱」**：每次覺得系統不夠好就砍掉重練，結果舊習慣被打斷、新流程還沒穩定，效率反而倒退
4. **漸進式擴展原則**：Unix 哲學的管線組合——每個工具做好一件事，超集只是把它們串起來，而非重新發明
5. **before/after 對比**：從「流程 A → 砍掉 → 流程 B」到「流程 A 不動 → 流程 B 包含 A + 新能力」，切換成本從斷裂變為零

## 頻道適合度
- ✅ **Threads**：核心概念本身就是一句話能講清楚的洞察——「新流程不取代舊流程，而是包含它」，適合用 1-3 則貼文帶出觀點，搭配軟體類比讓技術人秒懂
- ✅ **Blog**：最適合展開的頻道。可以寫完整的 before/after 實作案例、附程式碼或流程圖，從軟體 API 版本相容性切入，延伸到個人系統設計的通用原則
- ✅ **Facebook**：適合用「你有沒有過這種經驗——每次優化流程就要重新適應」的共鳴句開頭，帶出超集模式的概念，比 Threads 多一點敘事空間
- ❌ **Newsletter**：單獨撐一期電子報偏薄，但可以作為「本週洞察」的其中一個段落收錄，不適合獨立成篇
