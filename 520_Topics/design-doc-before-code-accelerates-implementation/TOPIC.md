# 先寫設計文件再寫 code 是加速器不是減速器

## 核心論點
把思考前置到設計文件，實作階段就變成純粹執行——21 分鐘 5 個 commit 不是因為寫得快，是因為寫之前已經沒有猶豫了。

## 關鍵素材
1. **實際數據**：21 分鐘完成 5 個 commit，從設計到交付全程無卡關，因為架構決策在文件階段已經定案
2. **反面對照**：不寫設計文件直接開幹的常見模式——邊寫邊改架構、反覆重構、一個 PR 拖三天，表面上「馬上動手」其實更慢
3. **認知負荷理論**：寫 code 時同時做架構決策，等於一邊開車一邊看地圖；設計文件讓你先規劃好路線，開車時只需要專注方向盤
4. **隱性成本**：沒有設計文件的專案，溝通成本藏在 code review 來回、PR comment 討論、甚至上線後的 hotfix 裡
5. **系列延伸素材**：開發者常見的「感覺浪費時間但其實省時間」習慣——寫測試、寫文件、做 code review、重構，都屬於同一類反直覺加速器

## 頻道適合度
- ✅ **Threads**：最適合。用「21 分鐘 5 個 commit」當 hook 數字感強烈，反差開場（你以為寫文件浪費時間？）正中 Threads 風格，150-250 字剛好講完一個觀點
- ✅ **Facebook**：適合。可以展開成故事——描述今天實際的開發過程，從寫設計文件到快速交付的完整體驗，結尾用開放提問（你寫 code 前會先寫設計文件嗎？）引發討論
- ✅ **Newsletter**：適合作為週報的一個段落素材。一句洞察 + 3 個 bullet（數據、對比、原理）的框架剛好裝得下，不需要獨立成篇
- ✅ **Blog**：可以延伸為教學文。加入設計文件模板、實際 before/after 對比、具體步驟，變成「如何用設計文件加速開發」的完整指南，SEO 潛力高
