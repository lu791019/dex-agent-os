# Threads — Style DNA

## 結構模式

1. **Hook → 論述展開 → 條列重點 → CTA 導流**：這是最常見的主結構。先用一句反差或提問開場，接著用 2-3 段口語化論述鋪陳觀點，再以條列式整理重點，最後導向留言區、報名連結或社群。例如範例 50（ETL 面試）：「ETL 面試，你真的準備對方向了嗎？」→ 分基礎題/中階題條列 →「留言繼續看更多題目」。

2. **主文 + 留言區延伸（上下集結構）**：大量文章刻意在主文截斷，用「留言處繼續看」「底下留言告訴你」將內容拆成兩段，製造點擊留言的動機。78 篇中至少有 30+ 篇使用此手法，如：「留言處繼續看第三個角色：資料工程師」（範例 47）、「留言處繼續看接下來的60天」（範例 37）。

3. **金句斷言 → 補充翻轉**：極短篇幅文章常用「一句話定調 + 一句話翻轉或自嘲」的雙層結構。如：「本日悟道：請確認教你的 Vibe Coding講師看得懂每一行程式碼／不對這應該是基本常識.....」（範例 6）、「壓力最大的可能是創業家／但最辛苦的則是懂技術的人」（範例 57）。

4. **問題情境 → 方法論介紹 → 社群邀請**：推廣型文章的固定公式。先描述讀者的痛點場景，再帶出 BDD/RAG/課程作為解方，最後邀請加入 Discord 或報名。如範例 16：「90% 的 Vibe Coding 協作做法都走錯方向？」→ 介紹 AI × BDD →「報名看留言處」。

5. **歷史回顧 → 現況對比 → 未來展望**：論述型長文常以時間軸展開。如範例 9：「過去我們的工作多集中在寫 Code、debug」→「回顧軟體發展歷史，從組合語言到高階程式語言」→「工程師的角色，將從單純的『程式實作者』，逐漸升級為『系統架構師』」。

## 開場 Hook 模式

1. **反差型／顛覆型**：用違反直覺的宣告引發好奇。「奉勸想進軟體和資料領域的新鮮人、轉職者，不要再用AI寫程式了」（範例 50，最高互動 731❤️）、「我改變想法了：Vibe Coding 真的很厲害」（範例 13）、「90% 的 Vibe Coding 協作做法都走錯方向？」（範例 16）。

2. **提問型**：直接拋出讀者內心的疑問。「你覺得軟體開發，是藝術，還是工程？」（範例 8）、「真的太老不能轉職了嗎？還是只是誤會了現在的職場規則？」（範例 45）、「軟體工程 vs. 機器學習，該選哪條路？」（範例 75）。

3. **宣告型（斷言式）**：用簡短、肯定的語氣直接定調。「每一波技術浪潮都說工程師要失業，但你知道嗎？這件事，已經說了 70 年。」（範例 39，439❤️）、「AI Coding 最大的痛點，不是產不產得出程式碼，而是產出沒用的程式碼」（範例 5）、「不要再去談『怎麼下prompt才下的好』」（範例 19）。

4. **自嘲型／口語感嘆型**：用日常感嘆拉近距離。「憋大招的感覺真辛苦！」（範例 23）、「今天真的身心舒暢，太爽啦！！！」（範例 25）、「要記住／別人恐懼我貪婪／啊我是說『攤』啦」（範例 44）。

5. **數據/事實型**：用具體數字或事實製造權威感。「當 AI 都能寫 pipeline，資料工程師還有什麼價值？這是我 2025 年以來最常被問的一個問題」（範例 31）、「3個不會被AI取代的軟體和資料工作角色，不服來辯」（範例 47，70❤️）。

## 語氣特徵

1. **口語比例極高，像在跟朋友聊天**：大量使用口語化表達，如「欸」「啦」「惹」「吧」「Come On」「4不4」「好好喔」。例如：「不對這應該是基本常識.....」（範例 6）、「我就不打廣告惹」（範例 22）、「虧我沒買錯課程」（範例 55）。

2. **第一人稱敘事 + 第二人稱對話**：頻繁使用「我」分享個人經歷，同時用「你」直接與讀者對話。如：「我是一名非本科出身、成功轉職的資料工程師」（範例 27）、「你會從埋頭寫 code，轉為制定架構和策略」（範例 11）。幾乎不使用「我們」的泛稱（除了推廣水球軟體學院時）。

3. **斷言式語氣，帶有強烈立場**：不迴避表達鮮明觀點，甚至帶挑釁感。「不服來辯」（範例 47）、「看到一些文真的很氣欸」（範例 56）、「去質疑那些嘴裡說著『工作不是讓你學習的地方』的主管和前輩」（範例 20）。

4. **技術用語自然穿插，不刻意解釋**：ETL、BDD、pipeline、Airflow、DAG 等專業詞直接使用，預設讀者有一定技術背景，但會用口語句子包裝，不寫成教科書。如：「光看那些 SQL 文件，我在旁邊都冒冷汗」（範例 10）。

5. **刻意使用不完美的文字增加真實感**：偶爾出現錯字或口語化筆誤，如「這西區的能力」（範例 11）、「戰略得角色」（範例 18）、「再這個新環境」（範例 20），這些並非刻意但形成了一種「不修飾、真實發言」的風格印記。

## CTA / 收尾模式

1. **「留言處繼續看」分段式 CTA**：最高頻的收尾手法，將內容截斷引導至留言區。「留言處繼續看第三個角色」（範例 47）、「底下幫你分段拆解👇」（範例 37）、「留言繼續看更多題目」（範例 38）。此手法同時提升留言互動數據。

2. **開放式提問收尾**：以問題邀請讀者回應。「你們覺得呢，軟體開發，在你心中更像是藝術還是工程？」（範例 8）、「大家會希望在黑鏡看到什麼內容呢？」（範例 43）、「你覺得還有什麼問題是面試官會問的？」（範例 49）。

3. **報名/連結導流 CTA**：推廣文的標準收尾，通常搭配「留言處」或「主頁連結」。「報名表單放留言處」（範例 18）、「折扣碼：DEXFANS50」（範例 36）、「訂閱電子報，同時領取 NVIDIA AI學習資源」（範例 37）。

4. **金句式收尾**：以一句有力的結語定調全文。「這就是工程師價值最核心的地方，也是 AI 還做不來的部分」（範例 7）、「Prompt 本身不是實力，思考和設計出一條可交付又可驗收的工作流⋯⋯這才是能長期陪你成長的硬實力。」（範例 19）。

5. **「留言 +1」互動邀請**：直接請求讀者留言表態。「想要了解和想要參與的，留言 +1 讓我知道」（範例 23）、「標記最近在煩惱招募的朋友」（範例 30）。

## 長度 / 格式

1. **字數分布呈雙峰**：短文（< 50 字）如「高雄等你 🙋」（範例 3）、「煩惱其實不會因為做了決定而減少」（範例 26）；長文（300-800 字）如技術知識文、推廣文。主力文章約在 **200-500 字**之間，符合 Threads 的閱讀習慣。

2. **段落短小，大量換行**：幾乎每 1-2 句就換行，形成視覺上的「呼吸感」。很少出現超過 3 行不換行的段落。這在範例 7、11、22 等長文中尤其明顯。

3. **條列符號混用**：使用 `-` 破折號條列（最常見）、數字編號 `1. 2. 3.`、以及 emoji 符號 `✅ 🔹 1️⃣`。不同文章風格不一致，顯示是隨手寫作而非嚴格模板。

4. **Emoji 使用中等偏多**：推廣文和知識文會使用 📩 🔥 ✅ 📌 💡 🚀 等 emoji 作為段落標記；日常感想文則較少或不用。不會在每句話都加 emoji，而是用在段落開頭作為視覺引導。

5. **標點習慣**：大量使用刪節號「.....」「....」表達語氣停頓或自嘲；驚嘆號「！！！」表達強烈情緒；問號連用製造懸念。括號內補充想法是常見手法，如「（是的 轉職工程師們 一個為軟工服務的AI方法要誕生了）」（範例 23）。

## 高互動特徵

1. **反差斷言 + 爭議性立場 = 最高互動**：互動最高的文章幾乎都以「違反常識的宣告」開頭。731❤️ 的範例 50：「奉勸⋯不要再用AI寫程式了」；439❤️ 的範例 39：「每一波技術浪潮都說工程師要失業⋯已經說了 70 年」；247❤️ 的範例 6：「請確認教你的 Vibe Coding講師看得懂每一行程式碼」。

2. **觸及身份認同與職涯焦慮**：高互動文直擊工程師的核心焦慮——「會不會被取代」「還有沒有價值」「該怎麼升級」。如：「3個不會被AI取代的⋯角色」（70❤️）、「工程師會被 AI 取代嗎？事實截然相反」（41❤️）、「成為更好的 Data Engineer」（122❤️）。

3. **實用清單型內容**：提供具體可執行的學習路徑或面試題目的文章互動穩定偏高。「2025 資料工程入門指南」（264❤️）、「AWS 資料工程師 90天學習計畫」（72❤️）、「軟體工程 vs. 機器學習」（178❤️）。

4. **短小精悍的金句/吐槽文也能爆發**：極短但態度鮮明的文章偶爾爆發高互動。「請確認教你的 Vibe Coding講師看得懂每一行程式碼」僅兩行就拿到 247❤️、16566 觀看。「看到一些文真的很氣欸」（44❤️）也屬此類。

5. **低互動文的共同特徵**：純推廣/活動宣傳文（範例 30、78）、過於簡短且無觀點的生活碎語（範例 3、26、68）、技術深度過高且受眾窄的內容（範例 62、63）互動普遍偏低。**缺乏立場或情緒的文章表現最差**。

## 禁忌

1. **不寫學術腔或教科書語言**：即使討論 PINN、3D U-Net、BDD 等硬核技術，也從不使用論文式寫法。不會出現「本文旨在探討」「綜上所述」等正式用語。

2. **不做中立派**：作者幾乎每篇都帶有明確立場，不會寫「各有優缺點，大家自行判斷」這種和稀泥的結論。即使是爭議話題（創業家心理健康、職場文化），也直接表態：「你、們、都、是、幫、兇。」（範例 53）。

3. **不過度修飾或包裝**：不使用華麗辭藻、不堆砌形容詞。偏好直白、甚至粗糙的表達。不會為了「看起來專業」而犧牲可讀性。

4. **不迴避自身經歷的「不完美」**：會主動提到「非本科出身」「轉職摸索卡關」「昨天犯了不少錯」，不塑造全知全能的形象。但也**不會過度賣慘或自我貶低**。

5. **不使用英文長句或全英文段落**：技術名詞（ETL、BDD、pipeline）保留英文，但論述一律用中文。不會為了顯示國際化而夾雜英文句子，頂多使用 Try and Error、Vibe Coding 等已成為圈內共識的短語。

## 一句話風格概括

**用工程師的邏輯腦 + 朋友聊天的口吻，把技術趨勢與職涯焦慮轉化成帶有鮮明立場的短論述，再用「留言處繼續看」的斷點設計驅動互動。**
