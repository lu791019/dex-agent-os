我在建 dex-agent-os 的過程中，串了十幾個服務的 API——Readwise、Fireflies、Google Classroom、Gmail——然後發現一件事：

決定一個服務能不能被 AI Agent 用的，不是價格，不是功能多寡，而是「它有沒有 API」。沒有就是死路，有就能串，沒有中間地帶。

這讓我開始想一個問題：API-first 的產品，第一批真正的客戶會是誰？

直覺會說大企業，因為預算多、需求複雜。但實際情況完全相反。

大企業要買一個 API 服務，得過採購流程、合規審查，最關鍵的是——非技術決策者需要「看到畫面」才能理解你在賣什麼。你跟他說「我們有一個 RESTful API，回應格式一致、錯誤處理優雅」，他會問你：「所以介面長怎樣？」

小團隊和獨立開發者完全不一樣。決策鏈可能就是一個人，打開 API 文件看十分鐘，跑幾個 curl 測試，確認能用就刷卡。對他們來說，API 文件就是產品頁面，回應格式就是 UI/UX。

我自己的體驗是：整合過程中最順的服務，不是功能最強的，而是 API 設計最乾淨的。文件清楚、格式一致、錯誤訊息有意義——這些在 AI Agent 時代就是產品力。

這帶出一個反直覺的結論：如果你在做 SaaS，與其一開始就瞄準企業客戶，不如先把 API 做好，讓獨立開發者和小團隊能無痛串接。這群人數量大、決策快、而且會幫你建生態系。

等 Agent 經濟真正起飛的時候，已經有開發者生態的產品，會比剛開始補 API 的產品跑得快很多。

你在串接各種服務的時候，有沒有遇過「功能很好但就是沒 API」的遺憾？或是哪個服務的 API 體驗讓你驚艷？
