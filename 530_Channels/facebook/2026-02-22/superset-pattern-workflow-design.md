每次覺得工作流程不夠好，你的第一反應是什麼？

我以前的反應是：砍掉重練。舊流程不好用？設計一個全新的。舊工具太陽春？換一套更完整的。聽起來很果斷，但結果往往是——舊習慣被打斷了，新流程還沒穩定，中間那段空窗期效率直接倒退。

後來我在設計自己的工作系統時，從軟體工程借了一個概念：backward compatibility。

做 API 的人都知道，出 v2 的時候不會直接砍掉 v1 的端點。v1 的行為會被保留，v2 只是在它之上加新能力。用 v1 的人不受影響，想升級的人自然遷移。這就是「超集模式」。

我把同樣的邏輯搬到個人工作流。

舉個例子：我原本有一個 `/daily-content` 的流程，專門處理每天的內容產出。後來需求變多了——要加同步、要加互動學習、要加自動萃取。我沒有把 `/daily-content` 改得面目全非，而是新建了一個 `/daily-all`，讓它包含 `/daily-content` 的所有功能，再加上新的步驟。

結果就是：忙的時候跑 `/daily-content`，有完整時間就跑 `/daily-all`。舊流程完全不動，新流程只是把它包起來。切換成本是零。

這跟 Unix 的管線哲學很像——每個工具做好一件事，超集只是把它們串起來，不是重新發明一個大工具。

回頭看，很多人（包括我自己）優化系統的方式其實是在做「大重構」：覺得不夠好 → 砍掉 → 從頭來。但真正可持續演化的系統，是讓新版本包含舊版本，而不是取代它。

你在設計自己的工作流程時，偏向「砍掉重練」還是「向上擴展」？有沒有踩過重構反而倒退的坑？
